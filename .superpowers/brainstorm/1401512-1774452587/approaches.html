AI Chat Panel UX — Three Approaches

Each approach addresses the same 5 requirements with different trade-offs

A

Incremental Refinement

Keep the current RichEdit-based architecture. Improve visual styling, add inline command blocks, and refine the thinking toggle. Lowest risk, fastest to ship.

Pros

  • Minimal structural changes to ai_chat.c
  • Reuses all existing RichEdit rendering
  • Can ship incrementally per-feature

Cons

  • RichEdit limits visual polish (no true message bubbles)
  • Thinking box remains a separate child window overlaid on display
  • Command blocks are text-only — buttons must still float outside
B

Hybrid: RichEdit + Inline Controls

Keep RichEdit for message text but embed lightweight child windows (buttons, command boxes) at calculated positions within the conversation flow. Thinking becomes a collapsible inline region.

Pros

  • Command approve/deny buttons appear right below their command block
  • Thinking box is inline and collapsible in real-time
  • Keeps proven markdown rendering in RichEdit

Cons

  • Positioning child windows inside RichEdit is fragile (scroll sync)
  • More complex layout logic in relayout()
  • Medium effort — more than A, less than C
C

Custom Owner-Draw Message List

Replace RichEdit with a custom-drawn scrollable message list. Each message is a discrete item with its own layout: bubbles, embedded thinking regions, command blocks with buttons. Full visual control.

Pros

  • Full control over every pixel — true ChatGPT-like feel
  • Each message is a self-contained unit with embedded controls
  • Clean architecture: message items instead of character offsets

Cons

  • Major rewrite — replaces the entire display system
  • Must reimplement text selection, scrolling, markdown rendering
  • High risk, longest timeline

💡 Recommendation: Approach B (Hybrid)

It gives you the inline command approval and real-time thinking toggle you need, without throwing away the proven RichEdit markdown rendering. The positioning complexity is manageable — you're already doing it with the thinking box. This approach extends that pattern to command blocks and approve/deny buttons.
